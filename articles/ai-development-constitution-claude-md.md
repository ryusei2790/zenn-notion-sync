「とりあえずやってみよう」で始めると、あとで詰む。

プログラミングを学びながら開発を進めていると、そう痛感することが増えてきました。設計をすっ飛ばして書いたコードは、後から読んでも何をしているのかわからない。そして何より、AIに「ちょっと直して」と頼むと、思ってもいない場所まで書き換えられてしまう。

そこで試してみたのが、Claude（AI）に「開発ルールの契約書」を渡す、という方法です。

## [CLAUDE.md](http://claude.md/) ってなに？

Claude Code というツールには、`CLAUDE.md` というファイルを読ませる仕組みがあります。

このファイルに「こういう開発フローで動いてほしい」「こういう指示には従わないでほしい」と書いておくと、AIがその通りに振る舞ってくれます。

私はこれを「開発憲法」と呼んでいます。

## 憲法の柱は4つ

### ① 曖昧な指示にはストップをかける

「ちょっと修正して」「なんか変にして」という抽象的な指示には、AIが実装を止めて質問を返すようにしています。

> 「指示が抽象的です。以下を明確にしてください」

と返ってくるように設定しています。

最初は少し面倒に感じました。でも、自分の指示の曖昧さに気づける機会が増えて、結果的に開発が速くなりました。

### ② 設計を承認してから実装に進む

Phase 0（問題定義）→ Phase 1（技術選定）→ Phase 2（設計）→ Phase 3（実装）という4段階のフローを設けています。

各フェーズで必ず「承認しますか？」という確認を挟み、OKが出るまで次に進めないようにしています。

最初は「回りくどい」と思っていましたが、設計段階でリスクを洗い出してもらえるようになってから、「作ったのに使えなかった」という無駄が格段に減りました。

### ③ コードを書いたら解説ドキュメントを自動生成

新しいファイルを作るたびに、こういう形式の解説ドキュメントが自動で生成されます。

```javascript
📘 ファイル解説：auth.ts

このファイルは何をするか
なぜこのファイルが必要か
データの流れ
初心者が詰まりやすいポイント
理解確認チェックリスト
```

「承認」と返信すると次のステップへ進む仕組みです。

これが思いのほか良くて、「コードは書けたけど何をしているかわからない」という状態がなくなりました。

### ④ 3つのドキュメントを自動で更新し続ける

開発中に以下の3ファイルが自動で更新されていきます。

これが一番気に入っています。開発を続けるほど「学習の証拠」が溜まっていく感覚があって、モチベーションが続きやすくなりました。

## 実際に使ってみて気づいたこと

正直に言うと、最初は「ルールが多すぎて逆に面倒では？」と思っていました。

でも使い続けてわかったのは、**このルールはAIへの縛りではなく、自分自身の思考整理のための仕組み**だということです。

「設計を承認してから進む」というルールは、「自分が本当にこれでいいか確認してから進む」ということでもあります。

AIに指示を出す前に、自分の頭の中が整理される。それが一番の変化でした。

## まとめ

- [CLAUDE.md](http://claude.md/) にルールを書くと、AIが「開発パートナー」として動いてくれる
- 曖昧な指示にはストップをかけてもらえる
- フェーズごとに設計 → 承認 → 実装 の流れを守れる
- コードを書くたびに解説ドキュメントが自動生成される
- 単語帳・プロジェクトログ・学習日記が自動で蓄積される
「AIをどう使うか」よりも「AIとどう開発するか」を考えるようになって、開発の質が上がった気がしています。参考になれば嬉しいです。
